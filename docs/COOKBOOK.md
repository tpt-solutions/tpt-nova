# TPT Nova Cookbook

Task-oriented recipes for common engine tasks. This complements the
reference-style [`GETTING_STARTED.md`](../GETTING_STARTED.md) with concrete,
copy-paste steps. Each recipe lists a **Goal**, **Steps**, **Code**, and
**See also** and is verified against the actual crate APIs (field names,
function names, and module paths were read from source).

> All snippets assume you are inside this workspace and `cargo` is on your PATH.

## Recipe: Add a new component + inspector field

### Goal
Make a new ECS component editable from the editor Inspector (and from the
agent/`set_field` path) by implementing `nova_editor::ComponentInspector` and
registering it in the inspector registry.

### Steps
1. Define your component (it must implement `nova_ecs::component::Component`).
2. In `crates/nova-editor/src/inspector.rs`, write a unit-struct inspector and
   `impl ComponentInspector for MyInspector` with three methods:
   - `name()` — the path prefix shown in the panel (e.g. `"Health"`).
   - `inspect(world, entity)` — snapshot the component's scalar fields as
     `ComponentInspection { component, fields }`, or `None` if the component is
     absent on `entity`. Each `Field.path` is dotted, e.g.
     `"Health.current"`.
   - `apply(world, entity, path, value)` — strip your prefix and write the
     matching scalar, returning `true` if recognized.
3. Register the type by adding `&MyInspector` to the slice returned by the
   `inspectors()` function. The panel and the `set_field`/`inspect_entity`
   functions stay generic — no editor code changes are needed beyond the
   registration entry.

### Code
```rust
// crates/nova-editor/src/inspector.rs

use nova_ecs::transform::Transform;
use nova_ecs::{Entity, World};

// 1) Your component (could live in nova-ecs or your own crate).
#[derive(Debug, Clone, Copy)]
pub struct Health {
    pub current: f32,
    pub max: f32,
}

// 2) Inspector for it.
struct HealthInspector;

impl ComponentInspector for HealthInspector {
    fn name(&self) -> &'static str {
        "Health"
    }

    fn inspect(&self, world: &World, entity: Entity) -> Option<ComponentInspection> {
        world.get_component::<Health>(entity).map(|h| ComponentInspection {
            component: "Health".to_string(),
            fields: vec![
                f("Health.current", h.current),
                f("Health.max", h.max),
            ],
        })
    }

    fn apply(&self, world: &mut World, entity: Entity, path: &str, value: f32) -> bool {
        if let Some(rest) = path.strip_prefix("Health.") {
            if let Some(h) = world.get_component_mut::<Health>(entity) {
                return match rest {
                    "current" => {
                        h.current = value;
                        true
                    }
                    "max" => {
                        h.max = value;
                        true
                    }
                    _ => false,
                };
            }
        }
        false
    }
}

// 3) Register it. `inspectors()` currently lives as a private fn in this file:
fn inspectors() -> &'static [&'static dyn ComponentInspector] {
    &[
        &TransformInspector,
        &RigidBody2DInspector,
        &HealthInspector, // <-- new entry
    ]
}
```

After editing, the existing `inspect_entity`/`set_field` functions pick up your
component automatically:
```rust
use nova_editor::inspector::{inspect_entity, set_field};

let mut world = World::new();
let e = world.spawn();
world.add_component(e, Transform::default());
world.add_component(e, Health { current: 80.0, max: 100.0 });

// Read it back through the generic inspector path:
let inspection = inspect_entity(&world, e);
assert!(inspection.iter().any(|c| c.component == "Health"));

// Write through the generic field path:
assert!(set_field(&mut world, e, "Health.current", 42.0));
assert_eq!(world.get_component::<Health>(e).unwrap().current, 42.0);
```

## Recipe: Add a new gameplay script

### Goal
Run your gameplay logic as a hot-reloadable Rust module, or as a sandboxed Rhai
script for AI-authored logic.

### Steps

**Rust hot-reload path (`nova-scripting` dylib).** The boundary is a small,
explicit C ABI — there is no `ScriptModule`/`GameScript` type; the trait is
`GameplayModule` and the exports are generated by the `export_gameplay!` macro:

1. Create a crate (copy `nova-gameplay-example`) whose `[lib]` is
   `crate-type = ["cdylib", "rlib"]` so it builds both a `cdylib` for the engine
   and an `rlib` for in-process unit tests.
2. Implement `GameplayModule` (it requires `Send` + `Default`) with at least
   `update(&mut self, ctx: &mut ScriptContext)`, plus optional `on_load` /
   `on_unload` hooks. `ScriptContext` gives you `world: &mut World`, `dt: f32`,
   and `tick: u64`.
3. Invoke `export_gameplay!(MyModule);` to emit the three ABI symbols —
   `_nova_gameplay_abi_version`, `_nova_gameplay_create`, `_nova_gameplay_destroy`.
   The host checks `ABI_VERSION` (currently `1`) before loading; a mismatch is
   rejected.
4. On the host side, load the built `.dll` / `.so` / `.dylib` with
   `HotModule::load(path)`. Each tick call `hot.update(&mut world, dt, tick)`. To
   get hot reload, call `hot.reload_if_changed(&mut world)` (the `HotModule`
   itself watches the file's mtime and swaps the library in place — there is no
   separate "watcher" object) before updating. The module is copied to a temp
   file on load so the original can be recompiled while running.

```rust
// In your gameplay crate (mirrors nova-gameplay-example/src/lib.rs):
use nova_ecs::transform::Transform;
use nova_ecs::{Entity, World};
use nova_scripting::{export_gameplay, GameplayModule, ScriptContext};

#[derive(Default)]
pub struct MyModule;

impl GameplayModule for MyModule {
    fn name(&self) -> &str { "my-module" }

    fn update(&mut self, ctx: &mut ScriptContext) {
        // Example: nudge every Transform up by dt.
        for (e, _) in ctx.world.query_1::<Transform>() {
            if let Some(t) = ctx.world.get_component_mut::<Transform>(e) {
                t.translation.y += 1.0 * ctx.dt;
            }
        }
        let _ = Entity::default(); // world/Entity imported, available if needed
    }
}

export_gameplay!(MyModule);
```

```rust
// Host side (e.g. your app or a test):
use nova_ecs::World;
use nova_scripting::HotModule;

let mut world = World::new();
let mut hot = HotModule::load("target/debug/libnova_gameplay_example.dll")
    .expect("load gameplay dylib");
hot.on_load(&mut world);

for tick in 0..120u64 {
    let dt = 1.0 / 60.0;
    hot.reload_if_changed(&mut world).ok(); // hot-swap if the file changed
    hot.update(&mut world, dt, tick);
}
```

**Rhai embedded path (`nova-scripting-embedded`).** Use this for AI-generated,
sandboxed logic. Scripts run in Rhai's safe interpreter with `eval`/`import`
disabled, and are further constrained by a capability boundary:

1. Build an `EmbeddedRuntime` with a `Capabilities` set. Only the functions for
   granted capabilities are registered (`spawn_entity`, `set_transform`,
   `log`, `emit_event`). Deny a capability and its function simply does not
   exist (the call is a caught error, not an escape).
2. `run(code)` (or `run_and_apply(code, &mut world)`) compiles and executes the
   Rhai source. Scripts never touch `World` directly; they enqueue typed
   `ScriptCommand`s into a queue.
3. `apply(&mut world)` drains that queue and applies the commands with full
   `&mut World` access. Read host logs with `take_logs()`.

```rust
use nova_ecs::World;
use nova_scripting_embedded::{Capabilities, EmbeddedRuntime};

let mut rt = EmbeddedRuntime::new(
    Capabilities::none().grant(nova_scripting_embedded::Capability::Spawn)
                        .grant(nova_scripting_embedded::Capability::WriteWorld)
                        .grant(nova_scripting_embedded::Capability::Log),
);
let mut world = World::new();
rt.run_and_apply(
    r#"
        let e = spawn_entity();
        set_transform(e, 1.0, 2.0, 3.0, 0.0, 0.0, 0.0, 1.0, 1.0, 1.0);
        log("spawned + positioned one entity");
    "#,
    &mut world,
).expect("script runs");
assert_eq!(world.entity_count(), 1);
```

Capability map (`Capability` → registered function):

| Capability   | Function(s) registered            |
|--------------|-----------------------------------|
| `Log`        | `log(msg)`                        |
| `Spawn`      | `spawn_entity() -> i64`           |
| `WriteWorld` | `set_transform(id, px,py,pz, rx,ry,rz, sx,sy,sz)` |
| `Net`        | `emit_event(name)`                |
| `ReadWorld`  | (reserved; no function yet)       |
| `Fs`         | (reserved; no function yet)       |

## Recipe: Ingest a custom mesh

### Goal
Load a `.glb` / `.gltf` / `.obj`, auto-generate a V-HACD convex-decomposition
collider and a procedural rig, then spawn it in the world as a simulated body.

### Steps
1. Put your model next to the committed sample `assets/cube.glb` (or pass an
   absolute path). Only `.glb`, `.gltf`, and `.obj` are supported
   (`IngestError::UnsupportedExtension` otherwise).
2. Call `nova_ingest::ingest(&path)`. This runs the whole chain:
   `load` → `decompose_meshes` (V-HACD) → `auto_rig` from combined bounds — and
   returns `IngestResult { meshes, convex_parts, skeleton, bounds }`.
3. Spawn an entity with `Transform` + `RigidBody3D` + `Collider3D::from_parts`
   using `result.convex_parts`. Add a `PhysicsWorld3D` resource and step with
   `nova_ingest::step_physics_3d(&mut world, dt)`.

```rust
use std::path::PathBuf;
use nova_ecs::transform::Transform;
use nova_ecs::{Vec3, World};
use nova_ingest::{ingest, Collider3D, PhysicsWorld3D, RigidBody3D};

// Zero-config: uses the committed assets/cube.glb.
let path: PathBuf = ["assets", "cube.glb"].iter().collect();
let result = ingest(&path).expect("ingest mesh");

let mut world = World::new();
world.add_resource(PhysicsWorld3D::default());

// Optional ground (large fixed convex slab).
let ground = world.spawn();
world.add_component(ground, Transform::from_translation(Vec3::new(0.0, -1.0, 0.0)));
world.add_component(ground, RigidBody3D::fixed());
world.add_component(ground, Collider3D::from_parts(nova_ingest::decompose::ConvexPart {
    vertices: vec![[-50.0,-0.5,-50.0],[50.0,-0.5,-50.0],[50.0,0.5,-50.0],[-50.0,0.5,-50.0],
                   [-50.0,-0.5,50.0],[50.0,-0.5,50.0],[50.0,0.5,50.0],[-50.0,0.5,50.0]],
    indices: vec![[0,1,2],[0,2,3],[4,6,5],[4,7,6],[0,4,5],[0,5,1],
                  [1,5,6],[1,6,2],[2,6,7],[2,7,3],[3,7,4],[3,4,0]],
}));

// The ingested mesh as a dynamic body, dropped above the ground.
let start_y = result.bounds.max.y + 5.0;
let mesh_entity = world.spawn();
world.add_component(mesh_entity, Transform::from_translation(Vec3::new(0.0, start_y, 0.0)));
world.add_component(mesh_entity, RigidBody3D::dynamic());
world.add_component(mesh_entity, Collider3D::from_parts(result.convex_parts.clone()));

for _ in 0..180 {
    nova_ingest::step_physics_3d(&mut world, 1.0 / 60.0);
}
```

You can also run the shipped zero-config demo:
```sh
cargo run -p nova-ingest --bin ingest_demo            # uses assets/cube.glb
cargo run -p nova-ingest --bin ingest_demo my_model.glb
```
To regenerate the only committed binary assets, run
`cargo run -p nova-ingest --example gen_sample_assets`.

## Recipe: Drive the engine from an AI agent

### Goal
Have an external process observe the world (telemetry) and mutate it (control
file) without restarting the engine.

### Steps
1. **Observe:** emit a `nova_telemetry::TelemetryFrame` each interval. Use
   `dump_world(&world, tick, seed)` plus a `TelemetrySink` (`FileSink`,
   `MsgPackFileSink`, or `StdoutSink`), or the interval-gated
   `TelemetryEmitter`. The agent reads the file (default name
   `nova-telemetry.json`) and extracts entity ids (`"e<index>#<generation>"`).
2. **Act:** build a `nova_agent_api::ControlFile` whose `protocol` equals
   `PROTOCOL_VERSION` (currently `1`) and whose `commands` are `AgentCommand`s
   (`Spawn`, `Despawn`, `SetTransform`, `SetRotation`). Serialize it with
   `to_json()` and write it to the agreed path.
3. **Apply:** on the engine side, a `ControlChannel::new(&path)` polls the file;
   `poll(&mut world)` re-applies only when the file's mtime changes, returns the
   number of commands applied, enforces the 1 MiB size cap, the
   `MAX_COMMANDS_PER_POLL` (10_000) cap, and the protocol-version check. The
   agent can also call `apply_command(s)` / `read_telemetry_file` directly.

```rust
use nova_agent_api::{AgentCommand, ControlChannel, ControlFile, EntityRef};
use nova_ecs::World;

let mut world = World::new();

// Agent writes a control file (here in-process; normally a separate process):
let cmd = AgentCommand::SetTransform {
    target: EntityRef::Id("e0#0".to_string()),
    translation: Some([2.0, 0.0, 0.0]),
    rotation_euler_xyz: None,
    scale: None,
};
let path = std::env::temp_dir().join("nova-control.json");
std::fs::write(&path, ControlFile::new(vec![cmd]).to_json().unwrap()).unwrap();

// Engine polls it:
let mut channel = ControlChannel::new(&path);
let applied = channel.poll(&mut world).expect("apply control file");
assert_eq!(applied, 1);
```

Observe side, minimal:
```rust
use nova_telemetry::{dump_world, FileSink, TelemetrySink};

let mut sink = FileSink::new("nova-telemetry.json");
sink.publish(&dump_world(&world, tick, seed)).unwrap();

// Agent reads it back:
let frame = nova_agent_api::read_telemetry_file("nova-telemetry.json").unwrap();
let hero = &frame.entities[0].id; // e.g. "e1#0"
```

## Recipe: Bind a live video texture to geometry

### Goal
Drive a mesh's surface from a streamed video/LLM feed using `nova-neural-materials`.

### Steps
1. Describe the feed with `MaterialPrompt::new(id, prompt, FeedSource)` and chain
   `.with_resolution(w, h).with_fps(f).with_tag(...)`. `FeedSource` is one of
   `VideoLlm { endpoint }`, `CaptureDevice(u32)`, or `File { path }`.
2. Create (or accept the default) `NeuralMaterialRegistry`, then
   `register(prompt)`. Each tick call `registry.update()` to pull the newest
   `Frame` from the provider. `MockProvider` synthesizes an animated gradient for
   tests/demos (no network needed).
3. Allocate a `NeuralTexture::create(device, w, h, label)` and each frame upload
   the latest frame: `registry.upload(id, &queue, &texture)` or, in custom code,
   `texture.upload(queue, registry.latest(id).unwrap())`. Sample it from a shader
   via `texture.view()`.
4. Record which entity is driven by which material with
   `MaterialBindings::bind(material_id, target_entity)`, then per frame pull
   `binding.latest_frame(&registry)` for rendering.

```rust
use nova_neural_materials::{FeedSource, MaterialBindings, MaterialPrompt,
                            NeuralMaterialRegistry, NeuralTexture};

let mut registry = NeuralMaterialRegistry::default(); // uses MockProvider
let prompt = MaterialPrompt::new(
        "billboard",
        "neon rain, cyberpunk",
        FeedSource::VideoLlm { endpoint: "llm://sora".into() },
    )
    .with_resolution(512, 256)
    .with_fps(24.0)
    .with_tag("cinematic");
registry.register(prompt).unwrap();

// Each simulation tick:
registry.update();
if let Some(frame) = registry.latest("billboard") {
    // `frame.width`, `frame.height`, `frame.rgba` (RGBA8) are ready to upload.
}

// Link the material to an on-screen entity:
let mut bindings = MaterialBindings::new();
bindings.bind("billboard", target_entity);
```

**PBR binding status (honest):** The contract is complete —
`FrameSource`/`NeuralTexture`/`NeuralMaterialRegistry` produce and upload live
frames, and `MaterialBinding` records "material `id` drives entity `target`".
The automatic wiring that pushes `registry.latest(id)` onto an entity's *PBR*
material in the renderer is **not yet implemented**: you must drive
`NeuralTexture::upload` and bind `view()` to your material yourself each frame
(the `binding.rs` module documents this as the "missing link" the renderer is
expected to fill in). `MockProvider` is the only built-in provider; swap in a
real Video-LLM client via `registry.set_provider(Box::new(MyProvider))`.

## Recipe: Save / load a scene

### Goal
Serialize the full ECS world to disk and restore it later, with version
migrations applied automatically.

### Steps
1. `nova_scene::save_to_file(&world, path)` — format is chosen by the file
   extension (`.ron` recommended, diff-friendly; `.json` also supported).
2. `nova_scene::load_from_file(path)` — parses, runs `migrate`, validates the
   scene graph (rejects duplicate ids and dangling parent/child refs), then
   re-instantiates entities with parent/child references remapped to fresh
   handles. `GlobalTransform` is derived after load, not stored.
3. For in-memory use, `dump_world(&world)` → `to_string(&scene, format)` and
   `from_str(&text, format)` → `load_world(scene)`.
4. Migrations: `CURRENT_SCENE_VERSION` is `2`. Loading a file newer than the
   current build errors with `SceneError::UnsupportedVersion`; older files are
   upgraded one version at a time by `migrate` (e.g. v1→v2 attaches a default
   `Light` to camera entities). Add a new arm to `migrate` when the schema
   changes; new component fields should be `#[serde(default)]` optional to stay
   backward compatible.

```rust
use nova_scene::{load_from_file, save_to_file, SceneFormat, dump_world,
                 from_str, load_world, to_string};

// To disk (extension picks the format):
nova_scene::save_to_file(&world, "my_scene.ron").unwrap();
let reloaded: World = load_from_file("my_scene.ron").unwrap();

// In-memory round-trip:
let scene = dump_world(&world);
let text = to_string(&scene, SceneFormat::Ron).unwrap();
let back: World = load_world(from_str(&text, SceneFormat::Ron).unwrap()).unwrap();
```

Supported component types in a saved `EntityRecord`: `Transform`, `Mesh`,
`Camera`, `RigidBody2D`, `Collider2D`, `Light` (v2+), plus `Parent`/`Children`
relationships.

## Recipe: Fork the sample game

### Goal
Start a new standalone project from the engine's reference template.

### Steps
1. Use the provided scaffolding helper (it copies `crates/nova-sample-game`,
   rewrites the crate name in its `Cargo.toml`, and prints next steps):
   ```sh
   pwsh scripts/fork_sample_game.ps1 -Name my-game
   ```
   This creates `./my-game` (a copy of the sample) with `name = "my-game"` in
   its `Cargo.toml`. Then:
   ```sh
   cd my-game
   cargo run
   ```
2. If you prefer a manual copy (or want the fork *outside* this workspace), copy
   the `crates/nova-sample-game` directory to your chosen location, edit its
   `Cargo.toml` `name`, and either add it to a Cargo workspace or run it
   standalone. Because `nova-sample-game` pulls in workspace crates by
   `version.workspace = true`, a standalone fork must either keep the workspace
   link or pin concrete versions/paths for the `nova-*` dependencies and fix any
   asset paths (the sample references `assets/cube.glb` relative to repo root).

The template (`nova-sample-game`) is the canonical end-to-end demo: physics,
scene save/load, agent control, splat ingestion, and asset packaging — a good
skeleton to delete bits from rather than start from an empty crate.

```sh
# Run the template itself (headless, no GPU):
cargo run -p nova-sample-game
```

### See also
- `scripts/fork_sample_game.ps1` — the scaffolding helper.
- `crates/nova-sample-game/src/lib.rs` (`run_pipeline`) and `src/main.rs`.
- `GETTING_STARTED.md` "Run the headless pipeline demo".
